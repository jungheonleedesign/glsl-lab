# 개념 노트 — GPGPU · 핑퐁 · semi-Lagrangian

이 셋은 **서로 다른 층**인데 같이 등장해서 헷갈리기 쉽다.
하나는 패러다임, 하나는 배관, 하나는 알고리즘이다.
(`FluidSim.tsx`, `FluidSolver2D.tsx`, `WaterParticles.tsx` 전부에 깔려 있음.)

## GPGPU — 패러다임

**General-Purpose computing on GPU.** GPU로 그래픽이 아니라 일반 계산을 한다.
GPU는 픽셀 수백만 개를 동시에 칠하는 거대 병렬 기계 → 이 병렬성을 아무 계산에나 빌려 쓴다.

WebGL엔 (전통적으로) 전용 컴퓨트 셰이더가 없어서 **렌더링을 계산으로 속여 쓴다:**

```
데이터를 텍스처에 저장        (배열 대신 텍스처)
풀스크린 쿼드 + fragment 셰이더 실행
   → 픽셀 1개 = 데이터 1개 병렬 계산
결과를 렌더 타깃에 출력        (그리기 = 계산하기)
```

우리 입자 물·유체 솔버가 전부 이 GPGPU. (WebGPU엔 진짜 컴퓨트 셰이더가 있어 이 둔갑이 덜 필요.)

## 핑퐁 — 배관 (HOW)

같은 텍스처를 읽으며 동시에 쓸 수 없다 → 버퍼 2장을 번갈아 쓴다.

```
read A → write B → swap   (다음 프레임엔 B가 read)
```

= '이전 상태'를 안전하게 들고 다니는 메커니즘. **상태/반복 계산을 가능케 함.**
무슨 계산을 하든 쓰는 범용 배관 (물결·라이프게임·입자·유체 다).

## semi-Lagrangian — 알고리즘 (WHAT)

이류(advection)를 푸는 한 가지 방법. "이 자리 값은 한 프레임 전 어디서 왔나?"

```glsl
vec2 src = uv - vel * dt;          // 흐름 거슬러 한 프레임 전 위치
float q = texture2D(uPrev, src);   // 거기 값을 끌어옴
```

GPU 픽셀은 자기 값만 쓸 수 있어 '밀어내기'가 안 됨 → '끌어오기'(역방향 추적)만 가능.

## 어떻게 맞물리나

```
┌─ GPGPU (GPU로 병렬 계산) ───────────────────────────┐
│                                                      │
│   핑퐁 (배관): read 버퍼 = 이전 프레임 상태           │
│      │                                               │
│      └─→ [semi-Lagrangian]                           │
│            src = uv - vel·dt                         │
│            read 버퍼에서 src 샘플 ← 이전 상태         │
│            결과를 write 버퍼에 → swap                 │
└──────────────────────────────────────────────────────┘
```

- **핑퐁**이 "이전 프레임"을 손에 쥐여주고,
- **semi-Lagrangian**이 그 이전 프레임의 어디를 읽을지(src) 정해 끌어온다.

배관이 재료를 대주고, 알고리즘이 그 재료를 쓰는 관계.

## 한 줄 정리

```
GPGPU            GPU를 일반 계산기로 씀. 패러다임.
핑퐁             버퍼 2장 번갈아 → 이전 상태를 들고 감. 배관 (범용).
semi-Lagrangian  흐름 거슬러 이전 상태를 샘플 → 이류. 알고리즘 (특정).
```

**핑퐁 ≠ semi-Lagrangian.** 핑퐁은 *어떻게 이전 상태를 들고 있나*,
semi-Lagrangian은 *그 이전 상태를 어떻게 쓰나*. 같이 다니지만 다른 층.
핑퐁 없이 semi-Lagrangian 불가(이전 상태가 없음), semi-Lagrangian 없이 핑퐁만도 가능(다른 계산).
